# AI 情绪指标 · 可靠性分析与重构方向

> 评审对象：股票详情页的「AI 情绪指标」模块（`server/routes/ai.ts` `/sentiment/:code` + `src/components/StockDetails.tsx`）。
> 依据：《选股交易操作手册》(`docs/选股交易操作手册.md`)。

## 一、当前实现

- **输入**：60 日 K 线 + 全量技术指标（MA/MACD/RSI/KDJ/BIAS/ATR/OBV/量比）原始数字。
- **LLM 任务**：扮演"趋势诊断师"，按 6 层框架（趋势/结构/时机/量价/风险/计划）**重新推导**技术分析。
- **输出**：`{score 0-100, label, summary, signals[], riskAssessment, tradePlan}`，缓存 4 小时。
- **前端**：与「量化交易信号」板块（确定性 `detectSignals` 引擎）**并列展示**，两套分数、两套信号列表、零交叉验证。

## 二、可靠性问题（五个）

### 1. 命名误用：它不是"情绪"
手册的**情绪**是**市场级情绪周期**（1.4：冰点→启动→加速→高潮→分化→冰点，由连板/炸板/换手接力判定），是 market-wide 的。本模块做的是**单股技术趋势诊断**，与"情绪"无关。命名误导用户预期。

### 2. LLM 重新推导确定性指标 —— 本质不可靠
手册的技术规则（MA 多头排列、MACD 金叉、RSI 超买…）是**确定性的**，应由代码计算。系统**已经**在 `signalEngine.detectSignals` 里忠实实现了（且与手册 3-6 章对齐）。让 LLM 从原始数字重新判断"MA5>MA10 是否成立"，是：
- **更不可靠**：LLM 的数值比较/算术易错；
- **不可复现**：同一数据不同调用/模型给出不同结论；
- **更贵**：把确定性工作交给按 token 计费的模型。
**LLM 在与确定性引擎做同一件事，且全面落败。**

### 3. 评分是黑箱
`score 0-100` 无定义的"规则→分数"映射，不同模型/日期对同一数据给出不同分。无法审计、无法回测。

### 4. 双轨冲突，用户无所适从
前端同时展示 `aiSentiment.score`（LLM）与 `signalReport.score`（引擎），两套信号列表，无一致性校验。当两者分歧（LLM 说"主升浪"、引擎说"空头排列"）时，用户不知该信谁。

### 5. tradePlan 价格是幻觉
LLM 给出的 `stopLoss/takeProfit` 具体价位是生成的，不如系统已有的 ATR/结构止损（`recommender.ts`/`cycles.ts`）可靠。

## 三、LLM 在这里真正的价值（手册交叉）

LLM 不该做**推导**，该做**解读与语义判断**——而手册里恰好有两类工作是确定性引擎做不到、LLM 适合的：

| 手册要点 | 引擎能否做 | LLM 价值 |
|----------|-----------|----------|
| 技术指标规则（3.x：MA/MACD/RSI…） | ✅ 已实现 | ❌ 不该重做 |
| **震仓 vs 出货**（6.5：打压后有无承接/资金护盘） | ⚠️ 部分信号，难全自动化 | ★ 语义判断 |
| **回调 vs 出货**（6.4：缩量/支撑/资金流） | ⚠️ 部分 | ★ 语义判断 |
| 把一串信号讲成"人话" | ❌ | ★ 叙事综合 |
| 情绪周期（1.4：连板梯队） | ✅ 信号引擎已加 N连板 | ❌ 不该重做 |

**结论**：AI 应定位为**确定性信号引擎的"解读层 + 语义判断层"**，而非平行的第二套推导。

## 四、重构方向（落地）

1. **正名**：「AI 情绪指标」→「**AI 诊股**」（趋势诊断，非情绪）。
2. **Grounding**：后端先跑 `detectSignals` 得确定性事实（分/排列/买卖信号/风险标签），**作为不可推翻的事实**喂给 LLM；LLM 只做综合叙事 + 震仓/出货语义判断，**不许重新推导指标**。
3. **一致性约束**：LLM 评分须在引擎分 ±10 内，否则标"AI 与引擎分歧"（暴露黑箱）。
4. **合并视图**：前端把"AI 情绪"与"量化信号"两板块合并——引擎事实为主干（可靠），AI 作解读层（叙事 + 震仓/出货判断 + 一致性徽章）。移除重复的 LLM 信号列表。
5. **诚实标注**：AI 分标注"解读"，引擎分标注"量化"；AI 的止损建议标注"参考，以 ATR 系统值为准"。

## 五、对照手册的覆盖

重构后，股票详情页对一个标的的评估将覆盖手册的核心维度：
- **技术信号**（3.x）：确定性引擎（可靠）。
- **震仓 vs 出货 / 回调 vs 出货**（6.4/6.5）：AI 语义判断（引擎盲区）。
- **情绪（连板）**（1.4/附录）：确定性连板信号。
- **风控位**：ATR/结构止损（已有）。
- **诊断叙事**：AI 综合（把以上讲成人话）。

## 六、闭环验证（2026-06-26，重构后实测）

`GET /api/ai/sentiment/sh600000`（调系统配置的 mimo-v2.5-pro）：
```
engineScore = 8  (引擎"弱势回避")     ← 可靠锚
aiScore     = 15 (AI"弱势震荡")        ← 解读，被约束在引擎 ±10 内
agreement   = { delta: 7, conflict: false }   ← 一致性可视（不再黑箱）
signals     = [缩量回调]  (引擎 buySignals，非 LLM 重推导)
sellSignals = [3天不创新高] (引擎)
summary     = "趋势中性偏弱，均线未形成多头排列。近期持续缩量回调，量价结构显示下跌动能衰竭..."
```
- AI 分被**约束在引擎分 ±10 内**（delta=7），消除了黑箱 ✓
- 展示的信号是**引擎可靠结果**，非 LLM 重新推导 ✓
- AI 提供叙事综合（基于引擎事实）+ 震仓/出货语义判断（引擎盲区）✓
- tsc + vite build 通过；dev server 实测正常 ✓

> 已知：部分模型偶发不返回 `diagnosis` 字段（如本次 null），前端优雅降级（不显示震仓/出货块）。架构正确，模型遵从度可后续用更强的 prompt 或模型提升。
